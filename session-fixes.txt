38. Upload Saga — ghost track: SetPublished ghi đè CANCELLED status
   - Vấn đề: Race condition — job bị cancel trong khi FFmpeg đang chạy:
     (1) Cancel request → DB status = CANCELLED
     (2) FFmpeg hoàn thành (chưa bị kill kịp) → TranscodeCompletedEvent published
     (3) Catalog nhận event → tạo Track record → publish TrackPublishedEvent
     (4) Upload nhận TrackPublishedEvent → SetPublished() → DB status = PUBLISHED
     Kết quả: job CANCELLED bị overwrite thành PUBLISHED, ghost track tồn tại trong catalog.
   - Root cause: 2 lỗ hổng độc lập:
     a. SetPublished SQL không check status hiện tại → overwrite bất kỳ status nào.
     b. transcoder.go không check xem job đã bị cancel trước khi publish event.
   - Fix (defense-in-depth — fix cả 2 lớp):
     a. adapter/postgres/job_repo.go SetPublished():
        WHERE id=$2  →  WHERE id=$2 AND status <> 'CANCELLED'
        (nếu status là CANCELLED → UPDATE ảnh hưởng 0 row → không overwrite)
     b. worker/transcoder.go process(), sau vòng lặp bitrate:
        if jobCtx.Err() != nil {
            log.Info().Msg("job cancelled during transcode, skipping event publish")
            return
        }
        (nếu job bị cancel trong khi một bitrate nào đó đang chạy → không publish event)
   - Lưu ý: Fix (b) là lớp phòng thủ đầu tiên — với per-job context từ fix #36, FFmpeg thường
     bị kill trước khi hoàn thành. Fix (a) là lớp phòng thủ thứ hai cho trường hợp FFmpeg đã
     hoàn thành trước khi CancelJob() được gọi (race condition timing). Cần cả 2.

KẾT QUẢ: Cancelled job giữ nguyên CANCELLED status; không có ghost track trong catalog.

39. Catalog service — ObjectOptimisticLockingFailureException khi ensureAlbumExists() chạy song song
   - Vấn đề: Upload album với nhiều track → catalog nhận nhiều TranscodeCompletedEvent gần như
     đồng thời → ERROR: "Row was already updated or deleted by another transaction for entity
     [Album with id '1ed62e53-...']" → track không được publish.
   - Root cause: ensureAlbumExists() dùng check-then-act pattern không atomic:
     a. existsById(albumId) → false (cả 2 transaction đều thấy false vì chưa ai commit)
     b. Cả 2 tạo Album entity in-memory với cùng albumId, gọi setId(albumId) → save()
     c. Spring Data JPA 4.x: isNew() check → ID đã set → false → gọi em.merge() thay vì persist()
     d. Tx1: merge() → SELECT → không thấy → INSERT → commit thành công
     e. Tx2: merge() → SELECT → thấy row (Tx1 đã commit) → Hibernate 7.2 phát hiện conflict
        giữa entity in-memory và DB row (AuditingEntityListener set updatedAt/createdAt khác nhau)
        → StaleObjectStateException → ObjectOptimisticLockingFailureException
   - Fix: Thay toàn bộ existsById() + new Album() + setId() + save() bằng native UPSERT:
     a. AlbumRepository: thêm insertIfAbsent() với @Modifying + @Query nativeQuery:
          INSERT INTO catalog_schema.albums (id, artist_id, title, cover_url, created_at, updated_at)
          VALUES (:id, :artistId, :title, :coverUrl, NOW(), NOW())
          ON CONFLICT (id) DO NOTHING
     b. AlbumService.ensureAlbumExists(): bỏ existsById() check + Album entity creation,
        thay bằng albumRepository.insertIfAbsent(albumId, artist.getId(), title, coverUrl)
   - Lưu ý: ON CONFLICT (id) DO NOTHING là atomic ở DB level — chỉ 1 trong N concurrent
     transactions insert thành công, các transaction khác silently skip (0 rows affected, không lỗi).
     Không cần check existsById() trước vì upsert đã handle idempotency.
     Đây cũng fix luôn trick setId() + merge() vốn là workaround brittle (fix #33).

KẾT QUẢ: Nhiều TranscodeCompletedEvent cho cùng album chạy đồng thời không còn conflict;
          album chỉ được INSERT một lần, các lần sau DO NOTHING.
